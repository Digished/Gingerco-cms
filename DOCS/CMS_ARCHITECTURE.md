# Ginger & Co CMS — Architecture

## Overview

The platform is a single Next.js 15 application powered by Payload CMS 3. It serves three things from the same deployment:

1. **Admin panel** — at `/admin`, for managing all content
2. **REST + Local API** — auto-generated by Payload for all collections
3. **Public website** — the frontend that renders content from the CMS

## Data Layer

### Database

Payload uses a PostgreSQL adapter (`@payloadcms/db-postgres`). Every collection and global maps to one or more database tables — Payload manages the schema automatically via migrations.

Migrations run automatically at the start of every deployment:
```
npm run payload -- migrate
```

To generate a new migration after changing a collection schema:
```
npm run payload generate:types
```
(Payload detects schema drift and creates migration files in `src/migrations/`.)

### Media Storage

All uploaded images go to **Vercel Blob** via the `@payloadcms/storage-vercel-blob` plugin. Payload's `sharp` integration generates resized variants automatically on upload.

If `BLOB_READ_WRITE_TOKEN` is not set (local dev without Blob), media falls back to local filesystem storage.

## Collections

Collections are defined in `src/collections/` and registered in `payload.config.ts`.

### Pages

Block-based pages. The `layout` field is an array of blocks. The frontend renders each block by matching its `blockType` to the corresponding React component in `app/(frontend)/components/blocks/`.

**Versions/drafts**: enabled. Pages can be saved as drafts and previewed before publishing.

### Events

Fitness classes, workshops, performances. Has `status` (draft / published / cancelled / completed), date, location, and a full block-based content area. Auto-saves drafts.

### BlogPosts

Article content with rich text (Lexical editor), author, cover image, and SEO fields.

### TeamMembers / Partners

Simple profile/listing collections. Used by the Team Block and Partner Section blocks respectively.

### Media

Image library. All uploads go to Vercel Blob. Payload generates `thumbnail`, `card`, and `hero` size variants via `sharp`.

### Subscribers

Newsletter subscribers. Fields:

| Field | Type | Notes |
|-------|------|-------|
| `email` | text | Required, unique, lowercase-normalised |
| `firstName` / `lastName` | text | Optional |
| `status` | select | `pending` → `subscribed` → `unsubscribed` |
| `confirmationToken` | text | Auto-generated hex token, used in email links |
| `unsubscribeToken` | text | Auto-generated hex token, used in email links |
| `subscribedAt` | date | Set when status changes to `subscribed` |
| `source` | text | Where they signed up (e.g. `homepage-footer`) |
| `tags` | array | For segmentation |

Both tokens are generated in a `beforeChange` hook using `crypto.randomBytes(32)`.

### EmailCampaigns

Newsletter campaigns. Fields include `subject`, `htmlBody`, `status`, `recipientFilter` (all / subscribed / tagged), `scheduledFor`, `sentAt`, `totalSent`.

The collection has a custom endpoint `POST /api/email-campaigns/:id/send` and a **Send Campaign** button injected into the admin edit view via `beforeDocumentControls`.

### Users

Admin users only. Payload handles authentication (cookie-based sessions). No public registration.

### Forms / Form Submissions

Managed by the `@payloadcms/plugin-form-builder` plugin. Forms are built in the admin with drag-and-drop fields. Each submission is stored in Form Submissions. Custom additions on top of the plugin's defaults:

- `arrivalNotice` — optional warning text shown at the top of the form
- `consentSections` — GDPR/waiver consent blocks with expandable legal text

## Custom Endpoints

Defined in `src/endpoints/` and registered in `payload.config.ts` under the top-level `endpoints` array. They are available at `/api/{path}`.

### `POST /api/subscribe`

Accepts `{ email, firstName?, lastName?, source?, tags? }`.

Flow:
1. Validates email format
2. Normalises to lowercase
3. If subscriber exists and is `pending` → resends confirmation email
4. If subscriber exists and is `subscribed` → returns success silently
5. If new → creates record (hooks generate tokens), sends confirmation email

### `GET /api/confirm-subscription?token=`

Flow:
1. Looks up subscriber by `confirmationToken`
2. If already `subscribed` → returns success page
3. Updates status to `subscribed`, records `subscribedAt`
4. Sends welcome email
5. Returns a styled HTML success/error page (not JSON)

### `GET /api/unsubscribe?token=`

Looks up by `unsubscribeToken`, sets status to `unsubscribed`, returns styled HTML page.

## Email System

All email is sent via the [Resend](https://resend.com) API.

**Two distinct usages:**

1. **Payload internal emails** (password resets, etc.) — configured via the `email` adapter in `payload.config.ts`
2. **Transactional subscription emails** — sent directly using the Resend SDK in endpoint handlers

**Templates** (`src/emails/templates.ts`):

| Function | Sent when |
|----------|-----------|
| `confirmationEmail()` | Subscriber signs up |
| `welcomeEmail()` | Subscriber confirms |
| `campaignEmail()` | Campaign is dispatched |

**URL variables used in emails:**

| Variable | Used for |
|----------|---------|
| `NEXT_PUBLIC_CMS_URL` | Building `/api/confirm-subscription` and `/api/unsubscribe` links — must point to the CMS deployment |
| `NEXT_PUBLIC_SERVER_URL` | "Explore Our Events" links back to the public website |

If `NEXT_PUBLIC_CMS_URL` is not set, it falls back to `NEXT_PUBLIC_SERVER_URL`.

## Build & Deployment

**Build command** (set in Vercel):
```
npm run payload -- migrate && npm run build
```

This runs pending migrations before the Next.js build. The build will fail if migrations fail (e.g. bad `DATABASE_URL`).

**TypeScript checking** runs as part of `next build`. All type errors fail the build — this is intentional.

**Import map** at `src/app/(payload)/admin/importMap.js` is auto-generated during build by `payload generate:importmap`. Do not edit it manually.

## Environment Variables

| Variable | Required | Purpose |
|----------|----------|---------|
| `DATABASE_URL` | Yes | PostgreSQL connection string |
| `PAYLOAD_SECRET` | Yes | Encrypts cookies and tokens |
| `RESEND_API_KEY` | Yes | Email sending |
| `RESEND_FROM_EMAIL` | Yes | Sender address |
| `NEXT_PUBLIC_SERVER_URL` | Yes | Public website URL (used in email links back to site) |
| `NEXT_PUBLIC_CMS_URL` | Yes* | CMS deployment URL (used in confirmation/unsubscribe links). Falls back to `NEXT_PUBLIC_SERVER_URL` if not set. |
| `BLOB_READ_WRITE_TOKEN` | Yes | Vercel Blob for media |
| `CRON_SECRET` | Auto | Auto-set by Vercel for cron jobs |
